# FAIR-in-action Playbook

A working **mockup** of the *FAIR-in-action* kit for the **BioFAIR Pathfinder — "The FAIR-in-action Bridge"** project. Built by the **ELIXIR-UK RDM Club**, commissioned by **BioFAIR**.

It is an [Astro](https://astro.build) static site that demonstrates the project's core idea — see [The thesis](#the-thesis).

---

## The thesis

Good FAIR *content* already exists in abundance (RDMkit, FAIR Cookbook, Galaxy, GO FAIR…). A pile of slides and scripts is just **another training course**. Our contribution is the **missing middle layer**: a way for an **RDM professional (the facilitator, not the researcher)** to

> **choose** the right activity for their audience and data type → **adapt** it to their institution → **deploy** it,

fast, while **curating and translating** existing resources rather than reinventing them.

Everything below serves that: one authored source per activity, rendered many ways, locally customisable without forking.

## One source → many outputs (the key idea)

Each activity is **authored once** as a `session:` block in a module's Markdown front-matter
(e.g. [`src/content/modules/02-public-repos.md`](src/content/modules/02-public-repos.md)).
From that single source we generate, with **no duplication**:

| Output | Where | Generated by |
| --- | --- | --- |
| **"Run this session"** steps on the activity page | `/modules/<id>/` | Astro ([`[...slug].astro`](src/pages/modules/%5B...slug%5D.astro)) |
| **Live present deck** (banded, profile-filled) | `/present/02-public-repos/` | Astro + reveal.js |
| **PDF handout** | "⬇ PDF" button → browser print | reveal print / page print CSS |
| **Editable slides (`.pptx`)** | download | [`scripts/session_to_pptx.py`](scripts/session_to_pptx.py) (python-pptx) |
| **Editable facilitator guide (`.docx`)** | download | [`scripts/session-to-decks.mjs`](scripts/session-to-decks.mjs) (Pandoc) |

Edit the `session:` once and **all five update together**.

### A `session:` step

```yaml
session:
  - title: "Finding the data"
    minutes: "15–25"            # cumulative range; UI shows the DURATION ("10 min")
    principles: ["F1", "F4", "A1"]   # facilitator reference only — hidden from researcher-facing slides
    objectives:                 # learning objectives (facilitator guide)
      - "..."
    tip: "..."                  # 💡 facilitator tip (facilitator guide)
    script: "..."               # 🗣 what the facilitator says → slide speaker notes + guide
    points:                     # on-slide bullets (inline HTML allowed)
      - "Accession <code>E-MTAB-9224</code> is persistent"
      - "Where they deposit: <span data-profile-repo>[ your repository ]</span>"
    exercise: "..."             # hands-on task → magenta callout
    image:                      # SVG/PNG; rendered on web, rasterised + embedded in decks
      src: "/assets/modules/02/img/components.svg"
      alt: "..."
```

The full schema lives in [`src/content.config.ts`](src/content.config.ts).

### Authoring a new activity (no code)

1. Add a Markdown file in `src/content/modules/` with the front-matter fields (copy [`_template.md`](src/content/modules/_template.md); files starting with `_` are ignored by the build).
2. Fill in the `session:` steps.
3. Commit. The deck generators + build run in CI and produce every format.

> Authoring is **Markdown only** — no developer needed for content. Only the *infrastructure* (templates, generators, theme) is developer-owned.

## Deck generators & tooling

Run locally (and automatically in CI):

```sh
node scripts/make-reference-templates.mjs   # build branded reference.docx/.pptx (run once / when brand changes)
node scripts/session-to-decks.mjs           # → <id>-facilitator-guide.docx  (Pandoc)
scripts/.venv/bin/python scripts/session_to_pptx.py   # → <id>.pptx  (python-pptx, banded)
```

Local tool prerequisites (already installed in this environment):

- **Pandoc** (`brew install pandoc`) — Markdown → `.docx`.
- **librsvg** (`brew install librsvg`, gives `rsvg-convert`) — rasterise SVG images for embedding.
- **Python venv** at `scripts/.venv` with `python-pptx` + `pyyaml` (`python3 -m venv scripts/.venv && scripts/.venv/bin/pip install python-pptx pyyaml`).

**Outputs** land in `public/assets/modules/<2-digit>/` and are copied into `dist/` by the build.

### Why two tools

- **`.docx` via Pandoc** — the facilitator guide is a *Word* document; Pandoc + a styled `reference.docx` (coloured callout boxes + section bands, injected by [`make-reference-templates.mjs`](scripts/make-reference-templates.mjs)) gets us a designed, editable guide.
- **`.pptx` via python-pptx** — to make the slides *look* banded **and** stay editable, we draw the bands/panels/callouts as native PowerPoint shapes (Pandoc only fills standard layouts).

### Fonts in downloads

Office files use **Calibri / Calibri Light** (ship with Word/PowerPoint) — Lato is **not** installed in Office and would fall back to Times New Roman. The **web** uses Lato. An institution can supply a branded `reference.docx`/`reference.pptx` (with Lato) — see [Branding](#branding).

## Runtime institution profile (customise without forking)

Forking goes stale the moment we ship a new module. Instead, an institution customises via a **thin profile stored in the browser** — one canonical, always-current kit.

- Set-up page: **[`/branding/`](src/pages/branding.astro)** — institution name, contacts, repositories, policies, **brand colours (hex)** and an **uploaded logo**.
- Stored in `localStorage` under `fair:profile`; helper `window.FairProfile` (get/set/clear) is defined in [`BaseLayout.astro`](src/layouts/BaseLayout.astro).
- On every page it **hydrates live**: the "viewing as" indicator, the `[ your repository ]` / local-support slots, the site **colours** (CSS variables) and the **header logo**. No fork; new activities appear automatically.
- A profile can be **exported/imported** as JSON to share across a team.

**Boundary:** the profile recolours the **web** live. Downloadable `.pptx`/`.docx` are generated ahead of time, so they carry the **canonical brand** (or an institution's uploaded **reference template**), not a colour typed into the browser afterwards.

## Deploy

[`.github/workflows/deploy.yml`](.github/workflows/deploy.yml) runs on push to `main`:

1. `npm ci`
2. Install Pandoc + librsvg + python-pptx, then run **both deck generators** (so downloads regenerate from source — never stale).
3. `npm run build`
4. Deploy `dist/` to **GitHub Pages**.

One-time GitHub setup: **Settings → Pages → Source: GitHub Actions**. Set `site`/`base` in [`astro.config.mjs`](astro.config.mjs) once the URL is known.

## Key decisions (rationale)

- **Shell, not content.** The differentiator is choose→adapt→deploy for facilitators, not more FAIR material. The audience is the **RDM professional**, not the researcher.
- **No fork; runtime profile.** A fork freezes and goes stale; the profile rides on one always-current kit. Forking is *allowed* but the set-up page is the default.
- **Curate, don't reinvent.** Each activity carries a `sources:` list ("Built from") — the value is translation + selection + local adaptation.
- **Single source → many outputs.** The `session:` block drives the page, live deck, PDF, `.pptx`, `.docx`. Edit once, everything updates.
- **Editable *and* designed downloads.** `.docx` via Pandoc + styled template; `.pptx` via python-pptx shapes. Office fonts stay Calibri-safe.
- **FAIR codes (F1/F4/A1…) are jargon to researchers** — hidden from researcher-facing slides/pages, kept in the facilitator guide.
- **Sustainability = non-dev ownership.** Content is Markdown the Club already writes; the developer-owned ~10% (generators, theme, CI) needs a named owner.
